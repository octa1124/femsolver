# `v0.3.0` Vector-Kernel Baseline

## Purpose

This document records the first implemented `v0.3.0` kernel slice that extends the completed scalar `v0.2.0` baseline toward the future motor magnetostatics path.

The goal of this step is not yet the full motor `A`-formulation. The goal is to establish the minimum self-owned `H(curl)` path that is still auditable:

- edge topology
- orientation-aware DoF numbering
- tetrahedral first-order `Nedelec`
- physical mapping for `H(curl)`
- curl-curl assembly
- one canonical vector benchmark

## What Is Implemented Now

The current repository adds the following concrete capabilities.

### Mesh Topology Extension

`kernel/mesh` now extracts unique tetrahedral edges and can identify boundary edges from boundary faces.

This is the first topology extension needed for edge-based spaces.

### Dedicated `kernel/space`

The new `kernel/space` module owns:

- node-based DoF maps for tetrahedral `H1`
- edge-based DoF maps for tetrahedral first-order `Nedelec`
- orientation-aware signed local-to-global DoF mapping
- boundary DoF extraction in solver numbering

The important boundary here is that basis functions remain local/reference objects, while `kernel/space` owns numbering and orientation.

### Tetrahedral First-Order `Nedelec`

`kernel/basis` now provides the first-order tetrahedral `Nedelec` family:

- six local edge basis functions
- basis evaluation on the reference tetrahedron
- constant reference curls

The basis functions follow the standard form:

- `N_ij = lambda_i grad(lambda_j) - lambda_j grad(lambda_i)`

where `lambda_i` are the barycentric coordinates of the reference tetrahedron.

### Physical Mapping

`ElementGeometry` now supports the two key mappings used by this first `H(curl)` slice:

- reference vector to physical vector through the inverse-transpose map
- reference curl to physical curl through the Jacobian-over-determinant map

This keeps the geometric transform explicit in repository-owned code.

### Curl-Curl Assembly

`kernel/assembly` now assembles:

- local tetrahedral curl-curl stiffness
- local vector-source load terms
- global edge-based sparse systems
- edge Dirichlet elimination

The current local stiffness is:

- `K_ij = integral(curl(N_i) dot curl(N_j)) dOmega`

The current local load is:

- `f_i = integral(J dot N_i) dOmega`

This is enough to validate the first edge-based finite-element path.

### Canonical Benchmark

`kernel/benchmark` now provides a manufactured curl-curl case on the centroid-refined reference tetra mesh.

The current exact field is a simple rotational vector field with exact edge moments. Boundary edge values are prescribed from that exact field, and the resulting linear system is solved with the existing self-owned conjugate-gradient path.

## What Is Not Implemented Yet

This current slice does not yet include:

- the full motor magnetostatic `A in H(curl)` formulation
- permanent-magnet source terms
- winding current-density source terms connected to machine regions
- gauge handling strategies beyond the fully prescribed benchmark case
- open-boundary treatment for motor outer air domains
- `H(div)` support
- hexahedral `H(curl)` support
- higher-order `Nedelec`

## Why This Slice Matters

The scalar `v0.2.0` baseline proved that the repository could own a mesh, basis, assembly, and benchmark path.

This `v0.3.0` slice proves the same idea for the first vector-field ingredients that the motor MVP will need:

- edges exist in the mesh layer
- edge orientation is explicit
- edge DoF numbering is explicit
- vector basis functions are explicit
- geometric transforms are explicit
- the global matrix path is still auditable

Without this slice, later motor work would risk collapsing back into opaque library dependence or application-specific shortcuts.

## Testing And Verification

Current automated coverage includes:

- edge extraction and boundary-edge tests in `kernel_tetra_mesh_test.cpp`
- DoF-map tests in `kernel_space_dof_map_test.cpp`
- basis interpolation and curl checks in `kernel_nedelec_basis_test.cpp`
- end-to-end vector benchmark verification in `kernel_curl_curl_benchmark_test.cpp`
- CLI coverage through `motor_check`

The current benchmark checks:

- solver convergence
- bounded iteration count
- exact recovery of edge DoFs within tight tolerance

## Current Risks

- the current quadrature path is still minimal
- the benchmark is canonical and controlled, not yet application-like
- the current vector source term is generic and not yet tied to motor region semantics
- edge Dirichlet handling is explicit and correct for the benchmark, but not yet optimized for larger systems

## Next Steps

The next meaningful extensions on top of this baseline are:

- connect case-region semantics to real motor source terms
- assemble the first magnetostatic motor smoke problem on `joint_type_i_12s10p`
- add richer vector-field validation cases
- add `Raviart-Thomas` for `H(div)` post-processing and future multiphysics exchange
- extend the same contracts to hexahedra and higher order without rewriting upper layers
