# `v2.0.0` Coupling Foundation

## Purpose

This document records the implemented `v2.0.0` multiphysics coupling foundation slice.

The goal is not to ship production thermal or mechanical physics yet. The goal is to stabilize the contracts that will let magnetostatics, thermal physics, and later mechanical physics share field state, assembly ownership, and analysis checks.

## Implemented Contracts

### `FieldState`

`FieldState` owns named field blocks.

Each block records:

- field name
- field kind
- unit string
- owned values

This is the first shared container for magnetic vector potential, temperature, displacement, and later generic fields.

`FieldState` also exposes stable field offsets and field sizes so block operators can embed local physics matrices into the coupled global system without guessing map order.

### `PhysicsOperator`

`PhysicsOperator` is callback-based rather than inheritance-based.

It records:

- operator name
- physics kind
- required field names
- an assembly callback

The callback returns a matrix, residual, and warnings. This keeps the extension point shallow and avoids locking the project into a deep class hierarchy before the coupled solver shape is proven.

### `CoupledProblem`

`CoupledProblem` owns:

- one `FieldState`
- a list of `PhysicsOperator` values

It assembles operator contributions into one coupled matrix and residual vector, while validating dimensions and required fields.

### Operator Adapters

`operator_adapters.hpp` adds the first concrete bridge from single-physics code into the coupled contract:

- `MakeMagnetostaticOperator` embeds a magnetic stiffness block and computes `K A - f`.
- `MakeMagnetothermalHeatOperator` adds a lumped heat residual driven by magnetic-field norm and a thermal sink.

The heat adapter is intentionally canonical rather than production thermal physics. Its job is to prove field offsets, cross-block Jacobian terms, residual aggregation, and warning propagation on a coupled magnetic-temperature state.

## Current Tests

The new test group verifies:

- named field blocks and total DoF accounting
- stable field offsets and sizes
- operator required-field validation
- multi-operator matrix and residual aggregation
- warning propagation through coupled assembly
- magnetostatic block embedding and residual evaluation
- magneto-thermal canonical residual and cross-Jacobian entries

The Python analysis tests also verify:

- machine report parsing
- completion-score calculation
- inheritance-depth checking
- version-readiness scoring from feature/test evidence

## Inheritance Policy

The current implementation deliberately uses composition and callbacks. The inheritance-depth checker in `tools/analysis/check_inheritance_depth.py` enforces the current preference that inheritance depth stays at or below `3`.

As of this implementation slice, the production C++ code does not rely on deep inheritance.

## Analysis Framework

`tools/analysis/analyze_machine_report.py` parses text output from `motor_solve` or `motor_check` and emits:

- raw parsed metrics
- named checks
- overall pass/fail
- completion score

This gives the project a non-visual analysis pipeline for local runs, CI logs, and future release summaries.

`tools/analysis/version_readiness.py` converts feature evidence into readiness scores for `v1.0`, `v1.1`, `v1.2`, `v1.3`, and `v2.0`. This prevents the project from treating a later-version contract as if the intermediate chain were already complete.

## What This Can Do

- assemble a coupled matrix/residual from multiple named physics operators
- embed a magnetostatic block into the global coupled system
- compute a canonical magneto-thermal residual with magnetic-to-thermal Jacobian entries
- keep coupled extensions callback/composition based with no deep inheritance chain
- produce non-visual readiness and result analysis for CI and release notes

## What It Cannot Do Yet

- time stepping and coupled nonlinear policies are not implemented yet
- production thermal conduction, boundary conditions, and material properties are not implemented yet
- production magneto-mechanical coupling is not implemented yet
- the machine `motor_solve` path still runs the linear magnetostatic solve directly rather than through `CoupledProblem`

Those are the next coupling milestones after the current `v2.0` foundation.
