# `v1.0.0` Solver Implementation Path

## Purpose

This document explains how the first real application MVP is expected to be implemented, which kernel milestones must happen before it, which algorithms are planned, and what remains out of scope.

It is the implementation counterpart to the product and physics documents.

## `v1.0.0` Goal

Deliver a self-owned solver that can compute a robot-joint permanent-magnet motor magnetostatic MVP with:

- magnetic vector potential `A`
- prescribed current-density windings
- permanent magnets
- magnetic energy and Maxwell-stress torque

## `v1.0.0` Explicit Non-Goals

- periodic sector infrastructure
- angle stepping
- sliding-band motion
- transient or eddy-current terms
- thermal, circuit, or NVH coupling
- broad mesh import and preprocessing flexibility beyond what the first machine cases need
- heavy reliance on MFEM or FEniCS runtime APIs in the production path

## Repository-Level Module Responsibilities

### `motor_pre`

Planned responsibility:

- read case configuration
- prepare or validate mesh and manifest inputs
- check region naming and output requests
- later host Gmsh-driven geometry and mesh preparation

At `v1`, this command is allowed to stay conservative. It does not need to become a full CAD environment.

### `motor_solve`

Planned responsibility:

- read case and mesh metadata
- construct finite-element spaces
- assemble residual and tangent
- run the nonlinear solve path
- produce field and scalar outputs

This is the production solver entrypoint.

### `motor_check`

Planned responsibility:

- run benchmark checks
- run regression checks
- later orchestrate canonical MFEM versus FEniCS comparisons

This command is the quality gate for solver behavior, not just a utility.

## Core Data Contracts

The following contracts should remain stable while the internals grow:

- `CaseSpec`
- `MeshManifest`
- material descriptors and material-law objects
- excitation definitions
- discretization strategy
- nonlinear policy
- solution bundle

They already exist in placeholder form and should be extended rather than replaced casually.

## Planned Solve Algorithm

### Stage 1: Input And Semantic Validation

- read the case definition
- load the mesh manifest
- verify region and boundary names
- verify which outputs are requested
- verify that the case uses only capabilities supported by `v1`

If the case asks for periodic sectors, transient behavior, or unsupported outputs, the program should fail clearly rather than silently degrade.

### Stage 0: Kernel Foundation Milestones

Before the motor MVP is credible, the repository needs self-owned kernel milestones:

- tetrahedral mesh and topology support
- reference tetrahedron and quadrature
- scalar `H1` basis and Poisson-style assembly
- sparse matrix assembly and boundary-condition handling
- edge orientation and `H(curl)` support

The motor MVP should build on those milestones rather than bypass them.

### Stage 2: Finite-Element Space Construction

- build the `H(curl)` space for `A`
- build `RT` or other auxiliary spaces as needed for magnetization and post-processing
- bind the mesh attributes to named physical regions

### Stage 3: Material And Excitation Binding

- construct region-wise material law objects
- bind current-density inputs
- bind permanent-magnet data in a local material frame
- prepare only the constitutive behavior supported by the MVP scope

### Stage 4: Residual And Tangent Assembly

Assemble the magnetostatic residual in terms of:

- the curl-curl contribution
- current-density excitation
- magnetization contribution
- constitutive response from the active material model

### Stage 5: Nonlinear Solve Path

Planned baseline algorithm for the MVP:

- start from a conservative initial state
- use the simplest stable solve path consistent with the material model in scope
- postpone fully nonlinear orthotropic solve complexity until the material-upgrade milestone

This path is chosen because it is simpler and more auditable than introducing a more elaborate trust-region method in `v1`.

### Stage 6: Post-Processing

- compute `B = curl(A)`
- integrate magnetic energy
- integrate Maxwell-stress torque on a predefined surface
- emit text summaries and field outputs

### Stage 7: Verification Hooks

- compare selected values to known benchmarks
- write compact regression summaries
- prepare data products for later `motor_check` comparisons

## Implementation Order

The recommended order is:

1. Real case parsing and mesh-manifest loading
2. Self-owned tetra mesh and scalar assembly on simple canonical cases
3. Self-owned `H(curl)` infrastructure and curl-curl assembly
4. Magnetization and current-density source terms
5. Robot-joint motor geometry and full-model case wiring
6. Post-processing and regression wiring
7. Nonlinear and anisotropic material upgrade in `v1.1.0`

This order keeps the risk concentrated in one new idea at a time.

## Test And Validation Plan For `v1`

### Unit-Level

- constitutive-law evaluation
- directional frame transforms
- tangent checks versus finite difference
- manifest and case validation

### Integration-Level

- simple canonical magnetostatic cases
- source-term smoke tests
- robot-joint motor full-model solve smoke test

### Regression-Level

- selected scalar outputs
- selected probe values
- field-export existence and metadata shape

## Why This Path Is Conservative

The first real motor MVP is already ambitious because it requires a self-owned `H(curl)` pipeline and a real machine geometry. The implementation therefore needs to be strict about scope control:

- no extra physics class in `v1`
- no extra geometry class in `v1`
- no extra solver family in `v1`

This is the shortest path to a defensible first machine solver rather than a broad but unreliable framework.
